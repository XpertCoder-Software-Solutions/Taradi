# Observability Runbook

## Logging

The backend uses Pino through `src/config/logger.js` and `pino-http`.

Required production settings:
- `NODE_ENV=production`
- `LOG_LEVEL=info` or `warn`
- `DEBUG=false`

Do not enable `DEBUG=true` in production unless investigating a short-lived incident, because debug webhook logs can include message text and operational payload details.

## Request Logs

HTTP request logs include:
- request id generated by `pino-http`
- method
- sanitized URL
- remote address and port
- status code and duration from `pino-http`

Sensitive request fields are redacted:
- authorization headers
- cookies
- password fields
- token fields
- configured WhatsApp/System token names

## Error Logs

Expected 4xx errors are logged at warn level. Unhandled 5xx errors are logged at error level with stack traces in logs only.

Production responses must not include stack traces. The current error middleware only includes stack traces when `NODE_ENV !== "production"`.

## Webhook Audit Trail

Every WhatsApp webhook request creates a `WebhookEvent` row before it is queued for worker processing. The HTTP handler should return quickly once the audit row and BullMQ job are created.
- provider
- event type
- WhatsApp message id when present
- sanitized headers
- payload
- processing status
- error message when failed
- processed timestamp

Important operational queries:

```sql
select status, count(*)
from "WebhookEvent"
where "createdAt" > now() - interval '24 hours'
group by status;
```

```sql
select id, "eventType", "whatsappMessageId", status, "errorMessage", "createdAt"
from "WebhookEvent"
where status = 'FAILED'
order by "createdAt" desc
limit 50;
```

## Queue Logs

Outbound WhatsApp jobs are handled by `taradi-whatsapp-worker`.

Inbound WhatsApp webhook jobs are handled by `taradi-webhook-worker`.

Log fields to watch:
- queue name
- job id
- message id
- customer id
- WhatsApp message id
- attempts made
- final status

Transient send failures are retried by BullMQ up to `WHATSAPP_QUEUE_ATTEMPTS`.

Webhook worker log fields to watch:
- queue name
- job id
- webhook audit event id
- event type
- WhatsApp message id, when present
- attempts made
- final webhook event status

Transient webhook processing failures are retried by BullMQ up to `WEBHOOK_QUEUE_ATTEMPTS`.

## Health Checks

`GET /health`

Use for basic process liveness. It does not check dependencies.

`GET /ready`

Use for production readiness. It checks:
- PostgreSQL query
- Redis ping
- outbound queue connectivity through Redis
- critical config status

Expected successful shape:

```json
{
  "success": true,
  "data": {
    "status": "ready",
    "service": "taradi-whatsapp-crm-backend",
    "checks": {
      "postgres": "ok",
      "redis": "ok",
      "queue": "ok",
      "webhookQueue": "ok",
      "config": "ok"
    },
    "queues": {
      "outbound": "whatsapp-outbound",
      "webhook": "whatsapp-webhook-processing"
    }
  }
}
```

## Metrics To Add Next

Recommended next production metrics:
- HTTP request count by route/status
- request duration histogram
- queue waiting/active/completed/failed counts
- webhook queue waiting/active/completed/failed counts
- webhook processing duration
- Meta API latency and error code counts
- media download/upload byte counts
- active Socket.IO connections by role
- unread conversation count by scope

## PII Handling

Avoid logging:
- full phone numbers
- national ids
- message bodies
- passwords
- tokens
- raw cookies
- full customer records

Phone numbers should be masked. `whatsapp.service` and `normalizePhone` already provide masking helpers in several paths.

## Incident Checklist

1. Check `/ready`.
2. Check PM2 status:

```bash
pm2 status
```

3. Check recent API and worker logs:

```bash
pm2 logs taradi-api --lines 200
pm2 logs taradi-whatsapp-worker --lines 200
pm2 logs taradi-webhook-worker --lines 200
```

4. Check failed webhook events.
5. Check BullMQ failed jobs and failed message rows.
6. Check Meta app status and token validity.
7. Disable campaign/template sends if Meta rate limits or token errors are active.
